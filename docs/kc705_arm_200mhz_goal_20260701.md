# KC705 ARM/Cortex-M0 + NPU 200MHz Goal Analysis

## 1. Scope And Assumptions

- Current scope is ARM/Cortex-M0 only. RISC-V/PicoRV32 is not part of this stage.
- The target FPGA board is KC705 with `xc7k325tffg900-2`.
- ILA must stay enabled. Do not use `-NoIla` and do not remove `ila_cpu_npu_kc705`.
- A bitstream with negative post-route WNS is not valid 200MHz signoff evidence, even if Vivado can generate it.
- No false path or multicycle path is added to the obfuscated Cortex-M0 internal logic without structural proof.

## 2. External Reference Findings

- AMD/Xilinx KC705 documentation identifies KC705 as a Kintex-7 evaluation board using the `XC7K325T` device family. The current Vivado part and board settings match the KC705 target.
- AMD Vivado timing-closure documentation treats post-route timing summary, critical path analysis, DRC/methodology reports, and QoR reports as signoff evidence. Therefore `WNS < 0` must be treated as a blocker for a 200MHz claim.
- AMD Vivado Hardware Manager and ILA flow supports loading a bitstream with its `.ltx` probe file, triggering ILA, uploading captured samples, and exporting waveform data. This matches the current `program_hw.tcl` and `capture_ila.tcl` automation path.
- Open accelerator projects and papers such as Gemmini, VTA, and NVDLA use the same architectural split that this design uses: CPU control, local memory or scratchpad, DMA/load-store movement, and a dedicated matrix/tensor compute array.
- These references support the architecture and validation method, but they do not replace local Vivado timing and board-capture evidence.

> For the ARM/Cortex-M0-only KC705 CPU + self-designed NPU system, reach a real 200MHz acceptance point in simulation and Vivado implementation: RTL regression keeps 1.024 TOPS@INT8, AXI Burst/DMA utilization is at least 80%, functional/path coverage passes; Vivado keeps ILA enabled and has a 5.000ns verification/data clock with final WNS >= 0 and no blocking DRC errors; then complete real-board program + ILA capture and produce a metric report.

## 11. 2026-07-01 Research And Code Analysis Addendum

This addendum records the current state after timing-closure attempts, external reference lookup, and a fresh code review. It is intentionally conservative: simulation metrics are counted as achieved only when backed by logs, and FPGA 200MHz is counted as achieved only when Vivado post-route timing is non-negative.

### 11.1 External Reference Summary

- AMD UG810 identifies KC705 as a Kintex-7 `XC7K325T-2FFG900C` evaluation board. This confirms that the current `xc7k325tffg900-2` Vivado part and KC705 board migration are the right FPGA target for this stage.
- AMD UG906 lists timing analysis, timing summary details, DRC, methodology checks, QoR reports, floorplanning, and implementation-closure techniques as the correct evidence chain for timing signoff. Therefore a `WNS < 0` bitstream is not acceptable as a 200MHz result.
- AMD UG908 documents the Vivado programming/debug flow, including bitstream download, Hardware Manager, ILA trigger/capture, waveform viewing, and saving captured ILA data. This matches the current `.bit/.ltx` plus `program_hw.tcl` and `capture_ila.tcl` direction.
- Gemmini, VTA, and NVDLA all support the architectural direction used here: a host CPU or control processor orchestrates a dedicated matrix/tensor accelerator, while local memory, DMA/load-store engines, and performance counters carry the data-plane work. These projects support the architecture choice but do not replace local RTL regression, timing, and board evidence.
- Gemmini is especially relevant because its public documentation describes a systolic array, scratchpad/accumulator SRAMs, DMA movement, runtime dataflow choices, and performance counters. The current project is smaller, but the same proof pattern applies: verify CPU control, accelerator execution, memory movement, and counters separately.

### 11.2 Current ARM-Only Code Facts

- CPU path: `rtl/cortex_m0_designstart_ahb.v` directly instantiates Arm DesignStart `CORTEXM0INTEGRATION`.
- SoC path: `rtl/cortex_m0_npu_soc.v` connects Cortex-M0 AHB-Lite to `ahb_lite_to_axil_bridge`, then to `axi_shared_interconnect`, `npu_accel_axi`, and `axi_bram`.
- KC705 strict top: `fpga/rtl/fpga_kc705_top.v` uses the board 200MHz differential clock and generates `soc_clk_mmcm` with `CLKOUT0_DIVIDE_F=5.000`, so the intended SoC/ILA clock is 5.000ns / 200MHz.
- ILA is retained through `KC705_ENABLE_ILA` and `ila_cpu_npu_kc705`; its probes include `npu_irq`, `npu_irq_latched`, `npu_array_clk_en`, `cpu_halted`, `debug_resetn`, `cycle_counter`, RAM beat counters, and DMA active/data/read/write counters.
- The local DesignStart package contains examples with an `SMUL` multiplier parameter, but the DesignStart obfuscated route explicitly does not pass parameters to `CORTEXM0INTEGRATION`. The current wrapper has no legal parameter hook for changing the internal Cortex-M0 multiplier or retiming the internal logic.

### 11.4 Root-Cause Assessment

- The NPU/AXI data-plane architecture is not the current strict 200MHz blocker. The known-good dual-clock implementation closes the NPU/AXI/ILA data plane at 200MHz.
- The strict single-clock blocker is the obfuscated `CORTEXM0INTEGRATION` internal logic on the KC705 `-2` speed-grade device. The worst path is not in user-editable NPU RTL.
- Prior attempts, including no-DSP multiplier macro use, Pblock variations, no-pblock routing, retiming, physical optimization, and post-route retiming, improved or changed symptoms but did not close the roughly 3.5ns negative slack.
- Fixing local warnings such as RAMB async control (`REQP-1839`) is still worthwhile for quality and robustness, but it is unlikely to recover the full 3.6ns single-clock Cortex-M0 slack by itself.
- Adding false paths or multicycle paths through Cortex-M0 internals would be unsafe unless there is structural proof from the processor implementation. No such proof exists in the current checkout.

4. Report route: the final report separates simulation performance estimates from board-captured activity evidence and cites the Vivado reports used for timing signoff.
